## 📚 Concepts Learned

### 1. Managed Identity (Digital ID Cards)
- **System-Assigned**: Tied specifically to one resource (like our Container App). If the app is deleted, the identity dies with it.
- **User-Assigned**: An independent resource that can be shared across multiple apps.
- **Why use it?**: It removes the need for "Secret Key" rotation and hardcoded passwords. The app just "is" who it says it is.

### 2. Azure Key Vault (The Safe)
- Centralized storage for secrets, keys, and certificates.
- **RBAC vs. Access Policies**: We used the modern **RBAC (Role-Based Access Control)** method. It is more scalable and integrates with standard Azure permissions.

### 3. Role-Based Access Control (RBAC)
- **Principal**: The "Who" (our App's Object ID).
- **Role**: The "What" (`Key Vault Secrets User`).
- **Scope**: The "Where" (our specific Key Vault).

### 4. Dependency Cycles & `-target`
- Learned that sometimes, Terraform can't calculate an ID until the resource is partially built.
- Used `terraform apply -target` as a safe "surgical" tool to build the Identity first.

## 🔥 Interview Questions & Answers

### Q1: What is the difference between an Access Policy and RBAC in Key Vault?
**A**: Access Policies are "Old School" and flat lists (permission to all or nothing). RBAC is the modern Azure standard that allows granular permissions (Secrets only, Keys only) and better auditing.

### Q2: Why did we experience a "Null" value error during the first plan?
**A**: Because the `principal_id` of a System-Assigned identity is generated by Azure *during* creation. Terraform didn't know the value yet, so we had to create the app first before we could use its ID in the Role Assignment.
